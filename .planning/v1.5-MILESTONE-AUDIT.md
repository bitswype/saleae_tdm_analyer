## Critical Gap: REQ-15

**One bug requires a code fix before v1.5 can ship.**

### BUG: Accumulate-before-flush ordering in `decode()`

**File:** `hla/TdmWavExport.py` lines 268–276
**Requirement:** REQ-15 — Each TDM frame = one WAV sample per channel

**Root cause:** In `decode()`, `self._accum[slot] = sample` executes BEFORE `self._try_flush(frame_num)`. When the first slot of a new TDM frame arrives (triggering a frame_number boundary), its data is written into the accumulator before the flush reads the accumulated values. The flush writes the new frame's data instead of the completed frame's data.

**Confirmed failure:**
```
Input:    TDM frame 1 → [slot0=100, slot1=200]
          TDM frame 2 → [slot0=300, slot1=400]

Expected: WAV [100, 200, 300, 400]
Actual:   WAV [300, 200, 500, 400]  ← slot0 always from next TDM frame
```

**Fix (one-line reorder):**
Move the accumulator update block (lines 268–270) to AFTER `_try_flush(frame_num)` (line 276):
```python
# CORRECT ORDER:
self._try_flush(frame_num)          # flush FIRST (reads clean accum)
self._accum[slot] = sample          # then accumulate new frame's data
self._last_frame_num = frame_num
```
